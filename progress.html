公开演示版

OSPF 混合仿真数字孪生实验平台

当前页面用于对外展示界面、项目边界和阶段进度。GitHub Pages 是静态展示环境， 控制台按钮使用浏览器本地模拟数据，不连接现场的 FastAPI、host-agent、OVS、Docker 或 QEMU。 真实实验仍在远端 Linux 主机上运行。

打开可交互控制台 查看 Demo 仓库

当前已完成

  • 前端控制台：拓扑编辑、运行验收、平台/RF 信息管理、状态提示和预检交互。
  • 演示模式：保存、规划、应用、下发、Ping、导出 YAML 均可在 Pages 上本地模拟。
  • 后端控制器：FastAPI API、实验快照、自动 IP/OSPF 规划、YAML 导入导出。
  • host-agent：在宿主机侧执行 OVS、Docker、QEMU、tap/veth 和清理流程。
  • x86 验证：Docker FRR/PC 与 QEMU NE40/H3C/Cirros 混合拓扑已完成基础连通性验证。

当前边界

  • Pages 展示版不会真正操作远端实验环境。
  • CLOUD/物理网卡接入需要现场独立网口和安全白名单，不能误接管理出口网卡。
  • 平台/RF 数据当前用于管理、校验、展示和导出，暂不影响 OVS 链路或 Ping。
  • QEMU qcow2 设备主要面向 x86/KVM，不能直接作为 ARM 通用能力承诺。

下一步重点

  • 补齐 ARM Docker-only profile 的真实主机验证。
  • 整理现场部署脚本、离线依赖包和迁移检查清单。
  • 完善 CLOUD 物理网卡接入的 UI 配置、风险提示和验收步骤。
  • 继续沉淀故障排查文档，降低无 AI 环境下的现场部署难度。

ARM 主机采购/测试需求

  • 必要性：目前 ARM 兼容性还停留在设计和 Docker-only 预期，缺少真实 ARM Linux 主机验收。
  • 用途：验证 Docker、OVS、host-agent、FRR/PC 容器、部署脚本和常见国产 Linux 环境适配。
  • 建议配置：ARM64，4 核以上，16GB 内存起步，128GB SSD 起步，Linux 可装 Docker 与 OVS。
  • 网口建议：至少 1 个管理网口；如果要测试 CLOUD/物理接入，最好有第 2 个独立网口。
展示版本：GitHub Pages 静态可交互 Demo。真实部署版本：controller Docker + host-agent + 宿主机 OVS/Docker/QEMU。
